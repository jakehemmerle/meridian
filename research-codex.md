# Meridian Research Rubric

This worksheet is for resolving requirement ambiguities before architecture, systems thinking,
and implementation planning. It is intentionally strict: every unresolved item should end with
either a decision, a documented assumption, or an explicit deferral with owner and deadline.

## How To Use This

1. Review each ambiguity in order.
2. Ask the listed questions in stakeholder or technical discovery sessions.
3. Record a decision, or if no decision is available, record the interim assumption.
4. Do not begin architecture or implementation planning for a row marked `Blocking` until it
   has a decision or an accepted interim assumption.

## Decision Rules

- Prefer primary sources: official protocol docs, oracle docs, and explicit stakeholder answers.
- If a requirement and an acceptance criterion conflict, escalate and resolve the conflict
  explicitly instead of averaging the two.
- UI-only constraints should not be treated as protocol guarantees unless the contract enforces
  them on-chain.
- If the product promises a user action, the contract and lifecycle must support it end-to-end.

## Status Legend

- `Blocking`: architecture cannot be finalized without an answer
- `High`: major design or scope impact, but work can proceed with a temporary assumption
- `Medium`: needed before detailed implementation, testing, or UX polish

## Architecture Gate

Do not finalize the architecture plan until these are resolved:

- `R1` Chain and execution venue
- `R2` CLOB approach
- `R3` Oracle availability
- `R4` Settlement source of truth
- `R5` Market-state transitions
- `R6` `Sell No` and pair closing

If any of the above remain unanswered, the architecture plan must explicitly state the temporary
assumption and the risk that assumption introduces.
